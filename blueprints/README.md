# Samourai Developer Kit Blueprints

A blueprint is a reusable model that describes how to produce a Samourai artifact: agent, skill, workflow, ticket, review, test plan, or policy. It acts as a generation contract for agents and a shared reference for development teams.

## Why Blueprints Exist

Blueprints industrialize repeated development-kit patterns without changing the existing runtime behavior. They help teams:

- standardize artifacts generated by agents;
- make required inputs explicit before generation;
- limit sensitive or destructive actions;
- keep templates short, reviewable, and versioned.

## How Agents Should Use Them

An agent must read the target blueprint `README.md`, then use the `*.blueprint.yaml` file as the generation contract and the Markdown template as the content base.

Example:

```text
Use blueprints/agents/agent.blueprint.yaml to create a release agent.
Render the result to .opencode/agent/release.md after human approval.
```

## Available Blueprints

| Blueprint | Purpose |
| --- | --- |
| `project-bootstrap/` | Initialize project instructions and Samourai conventions. |
| `agents/` | Generate an OpenCode-compatible agent. |
| `skills/` | Generate a reusable operational skill. |
| `workflows/` | Describe a team or agent workflow. |
| `code-review/` | Structure code review criteria and findings. |
| `jira/` | Draft an actionable Jira ticket. |
| `github/` | Prepare a GitHub PR and review. |
| `debugging/` | Produce a diagnostic report. |
| `testing/` | Produce a test plan. |
| `security/` | Draft a security policy or checklist. |

## Naming Conventions

- Directories use kebab-case; plural names are used for reusable collections (`agents`, `skills`, `workflows`).
- YAML contracts use `<name>.blueprint.yaml`.
- Markdown templates use `<name>.template.md` or a specific artifact name such as `code-review-checklist.md`.
- Variables use `<placeholder>` syntax.
- Generated artifacts should reference their source blueprint when that helps maintenance.

## Security Rules

- Never add, read, display, store, or copy secrets, tokens, API keys, cookies, passwords, or credentials.
- Treat GitHub, Jira, CI/CD, publishing, merge, deletion, rebase, and ticket transitions as sensitive actions.
- Propose and explain before running any destructive or external action.
- Explicitly list allowed tools in every agent blueprint.
- Keep temporary artifacts under `.samourai/tmpai/` and local state under `.samourai/ai/local/`.
- Never bypass human approvals required by the workflow.
