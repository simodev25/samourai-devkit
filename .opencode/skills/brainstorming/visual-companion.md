# Visual Companion

Practical guide for adding visual artifacts during brainstorming sessions.

## When to Offer Visuals

Offer visual support when the discussion involves:
- UI layouts, navigation, or screen hierarchy
- Component composition and spacing
- Flow diagrams (multi-step user journeys)
- Trade-offs that are easier to compare side-by-side

Keep text-only when questions are conceptual, scope-related, or purely technical.

## How to Describe UI Concepts

Use a consistent structure:
1. **Context** — where this appears (page, modal, wizard step)
2. **Intent** — what user goal it supports
3. **Structure** — regions/components from top to bottom
4. **Interaction** — key states, transitions, and actions
5. **Constraints** — responsiveness, accessibility, performance limits

Suggested phrasing:
- “Top area shows primary status and next action.”
- “Main panel contains editable fields grouped by task.”
- “Right rail is optional contextual help on large screens only.”

## Visual Artifact Formats

Choose the lightest artifact that answers the question:

- **Wireframe (low fidelity):** layout and hierarchy only
- **Annotated mockup:** adds labels for behavior and states
- **Flow diagram:** decisions, branching, and transitions
- **Comparison board:** Option A vs B vs C with pros/cons

For each artifact, include:
- Title and objective
- Version/date
- Assumptions
- Open questions

## Quality Checklist

- Is the visual answering a specific decision?
- Are labels unambiguous and readable?
- Are key states represented (empty/loading/error/success)?
- Is responsive behavior explained for mobile + desktop?
- Are accessibility notes captured (focus order, contrast, keyboard)?
